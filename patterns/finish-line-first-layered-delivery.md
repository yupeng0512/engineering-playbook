# 先定义终局，再分层推进（Finish-Line-First Layered Delivery）

## 问题

复杂治理、架构、流程或工具链类工作，天然会不断冒出“下一步”。

如果一开始不明确：

- 最终目标是什么
- 当前这次要做到哪一层
- 做到什么算完成
- 后续哪些只是增强项

那么协作就会逐渐滑向一种危险状态：

- 每次都像“还差最后一步”
- 用户无法判断现在是不是已经够用
- Agent 会不断给出下一步建议，但边界越来越模糊
- 项目进入一种没有终点感的持续打磨

## 核心原则

> 先定义终局，再定义当前层；先讲完成线，再讲增强项。

这类工作不应该只汇报“下一步是什么”，而应该同时汇报：

1. 最终目标
2. 成熟度分层
3. 当前目标层
4. 完成标准
5. 可选增强顺序
6. 每个下一步的必要性与收益

## 适用场景

- 发布治理
- 架构治理
- 测试体系建设
- Agent workflow / autonomy contract
- 运维自动化体系
- 长链路体验优化

凡是“天然可以不断继续优化”的工作，都适合用这个模式。

## SOP

### Step 1: 先写最终目标

不要从“下一步做什么”开始。

先写一句话回答：

- 这条线最终要解决什么问题？
- 什么状态下可以说“这条线已经建立起来了”？

例子：

- 发布治理的终局不是“更多 gate”，而是“每次发布都知道风险、验证、回滚和 preview 边界”

### Step 2: 把目标拆成成熟度层

把工作拆成 `3-6` 层最容易理解。

每一层都应该是一个用户能感知到的能力，而不是一个零散动作。

好的分层方式：

- Layer 1: 基础能力存在
- Layer 2: 原则被写清楚
- Layer 3: 能执行
- Layer 4: 进入日常流程
- Layer 5: 平台级强制执行

### Step 3: 明确“当前停线”

必须明确回答：

- 当前阶段打算停在哪一层？
- 为什么停在这里是合理的？

这一步非常关键。

没有“停线”，就没有“完成”。

### Step 4: 区分“本次完成”与“后续增强”

收尾时必须把两者分开说：

- `本次目标是否完成`
- `如果继续升级，下一层成熟度是什么`

不能把“未来增强建议”说成“当前仍未完成的缺口”。

### Step 5: 写清楚“为什么是这一步”

一旦进入“后续增强”区，不能只写动作顺序。

每个下一步都应该回答三件事：

- **为什么现在做这一步**
- **它解决的核心约束是什么**
- **它带来的收益是什么**

推荐从第一性原理来写，而不是从“顺手”或“技术上还能继续做”来写。

最低要求至少写清：

- `Why now`
- `Expected benefit`

如果是高成本/高风险升级，最好再补：

- `If skipped now, what remains acceptable`

### Step 6: 用一页路线图固化

对项目内至少沉淀一页短文档，明确：

- 当前已经完成到哪
- 现阶段可以停在这里
- 未来升级顺序
- 每个升级顺序背后的必要性与收益

这份文档的作用不是详细设计，而是建立边界感。

### Step 7: 把执行入口接回日常流程

如果这条线已经有可执行动作，再把它接入实际工作入口：

- checklist
- PR template
- runbook
- Makefile
- release gate

也就是说：

- 路线图负责“说明边界”
- checklist/template 负责“进入日常使用”

## 收尾汇报模板

推荐至少用这五段来收尾：

```md
## Final Target

[一句话说明终局]

## Current Layer

[当前做到第几层，哪些能力已经有了]

## Stop Line

[为什么现在可以停在这里，这次已经完成]

## Future Upgrades

[后续如果升级，按什么顺序升级；这些属于增强项，不属于本次未完成项]

## Why This Next

[每个下一步为什么要做、解决什么约束、预期收益是什么]
```

## 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 每次只说“下一步” | 用户不知道当前是否已完成 | 先说当前层是否完成 |
| 只写步骤，不写收益 | 用户不知道为什么值得继续投资源 | 每个下一步都写 `Why now + Expected benefit` |
| 一边做一边临时扩目标 | 范围失控 | 先写终局和分层 |
| 把增强项说成缺口 | 永远没有完成感 | 明确区分“已完成”与“可升级” |
| 只有策略没有执行入口 | 文档好看但不会被用 | 配 checklist / template / runbook |
| 只有执行动作没有边界说明 | 团队不知道什么时候该停 | 用一页路线图说明停线 |

## Definition of Done

这一模式被正确应用时，应至少产出：

- [ ] 一个明确的最终目标描述
- [ ] 一份成熟度分层
- [ ] 一个清楚的当前停线
- [ ] 一份项目内的一页路线图
- [ ] 每个下一步的必要性与收益说明
- [ ] 至少一个进入日常流程的执行入口

## 例子

### Case: TradeRadar 发布治理

正确做法：

- 先定义终局：发布风险、preview、验证、回滚都必须清楚
- 再分层：自动化发布 → 策略 → checklist → PR gate → 平台硬门禁
- 明确当前停线：先停在 PR gate 这一层
- 再说明后续增强：GitHub environment approval、temporary preview、formal preview gate

这样用户就能知道：

- 当前这条线已经完成
- 后续建议是增强项
- 不是“永远还有下一步所以永远没做完”
