# Pattern: 遇到问题先调研（Research Before Solving）

> 来源: trading-system Polymarket API 网络问题的反思
> 创建: 2026-03-01
> 适用于: 任何技术问题、架构设计、工具选型

## 问题

遇到技术问题时（如 API 连接失败、SSL 错误），直接假设原因并尝试修复，
导致方向错误（如错误地认为需要代理，实际是内网防火墙问题）。

## 核心原则

> **你遇到的问题，99% 概率别人也遇到过。先站在巨人的肩膀上。**

## 解决方案：三步调研法

### Step 1: 查官方文档

在尝试任何修复之前：
1. 找到问题相关的**官方文档**
2. 确认自己的理解是否正确（如 API 是否真的被封禁？）
3. 检查是否有已知的限制、geoblock、rate limit 等

**反面案例**: Polymarket SSL 连接失败 → 直接建议配置代理
**正面案例**: 先查 Polymarket Geoblock 文档 → 发现中国不在封禁名单 → 确认是内网问题

### Step 2: 搜索开源项目

搜索 GitHub 上同领域的高星项目，看他们怎么解决同类问题：
- `github [项目名] [问题关键词] [语言] 2025 2026`
- 看他们的 `.env.example`、README、issue 讨论
- 重点关注：网络配置、认证方式、部署架构

**收获**: 不仅解决当前问题，还能发现更好的架构模式

### Step 3: 搜索社区讨论

- Stack Overflow / GitHub Issues / Reddit / Twitter
- 搜索具体的错误信息
- 看是否有 workaround 或官方回复

## 调研输出模板

每次调研完成后，沉淀到项目的 `docs/research/` 目录：

```markdown
# [问题/主题] 调研报告

> 调研时间: YYYY-MM-DD
> 问题: [一句话描述]

## 真因分析
[问题的根本原因是什么？]

## 已有方案对比
| 方案 | 来源 | 优劣 |
|------|------|------|

## 对我们的建议
[基于调研结果的行动项]
```

## 适用场景

| 场景 | 调研重点 |
|------|----------|
| API 连接失败 | 官方文档 → geoblock/rate limit → 开源项目的网络配置 |
| 架构设计 | 同领域高星项目 → 技术选型 trade-off |
| 工具选型 | 对比 3+ 个同类工具 → 社区活跃度 → 维护频率 |
| 性能问题 | 官方 benchmark → 同规模用户的经验 |
| 安全问题 | CVE 数据库 → 官方安全指南 → 漏洞修复方案 |

## 反模式

| 反模式 | 后果 | 正确做法 |
|--------|------|----------|
| 直接猜原因并修复 | 方向错误，浪费时间 | 先查官方文档确认 |
| 只搜一个关键词 | 遗漏关键信息 | 多角度搜索（官方 + 社区 + 开源） |
| 调研完不沉淀 | 下次遇到同样问题重新来过 | 写 `docs/research/` 报告 |
| 闭门造车 | 重复发明轮子 | 先看业界最佳实践再动手 |

## Checklist

遇到问题时按此顺序执行：

- [ ] 1. 找到并阅读**官方文档**中的相关章节
- [ ] 2. 搜索 GitHub 上 **3+ 个高星开源项目**的处理方式
- [ ] 3. 搜索社区中的**类似问题讨论**
- [ ] 4. 形成调研结论，沉淀到 `docs/research/`
- [ ] 5. 基于调研结果制定解决方案

## 案例库

### Case 1: Polymarket API SSL 连接失败

- **表象**: `ConnectError: [SSL: WRONG_VERSION_NUMBER]`
- **错误假设**: Polymarket API 需要代理
- **调研发现**:
  1. 官方 Geoblock 文档: 中国不在封禁名单
  2. Gamma API / Data API 完全公开，无需认证
  3. 开源项目 (Gabagool): 数据获取不需代理，下单才可能需要 SOCKS5
  4. SSL 错误特征指向内网代理/防火墙拦截 HTTPS
- **正确方案**: 确认内网是否有透明代理 → 配置直连或换环境
- **额外收获**: 发现了多个高星开源项目的架构模式，完善了跟单策略
