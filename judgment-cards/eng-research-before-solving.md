# 判断力卡片: 遇到问题先调研

## 命题
遇到技术问题时，先在官方文档和开源社区中调研已有方案，再动手解决。你遇到的问题 99% 概率别人也遇到过。

## 领域
`eng` — 工程管理

## 推导
1. 技术问题的解空间巨大，错误假设会导致方向性错误
2. 互联网上存在大量同类问题的解决方案和讨论
3. 高星开源项目经过社区验证，方案可靠性远高于个人猜测
4. 调研本身也是知识积累，常有意外收获（新工具、新模式、新思路）

## 三步法
1. **查官方文档** — 确认问题本质，避免错误假设
2. **看开源项目** — GitHub 上 3+ 个高星项目的处理方式
3. **搜社区讨论** — Stack Overflow / Issues / Twitter 中的经验

## 成立条件
- 问题涉及外部系统、API、工具链
- 问题非项目内部的特定业务逻辑 bug
- 有互联网搜索能力

## 失效条件
- 问题是首创性的，确实无前人经验（极罕见）
- 紧急生产故障，需要立即止损（先救火，后调研）

## 验证案例
Polymarket API SSL 连接失败。直接假设「需要代理」并添加代理配置是错误的。
调研发现：中国大陆不在 Polymarket Geoblock 名单中，SSL 错误实为内网防火墙问题。
额外收获：发现多个高星开源项目的架构模式，完善了系统设计。

## 反模式
- 「直接试试看」→ 错误假设导致方向偏差
- 「我来写一个」→ 重复发明轮子
- 「调研完就算了」→ 不沉淀，下次重来
