## 三、具体优化建议

### 需要行动的项目（4 项）

#### 1. `flake8.mdc` → 精简（C 级，5.65 分）

**问题**：276 行过于通用的 Python 最佳实践百科，与网上随处可查的 Flake8 文档高度重叠。作为 `alwaysApply` 规则，每次对话都注入 276 行通用内容，性价比极低。

**建议**：精简至 50-80 行，仅保留：
- 项目特定的 `.flake8` 配置项及其"为什么这样配"的说明
- 本项目实际踩过的 Flake8 坑（如特定 rule 的 ignore 理由）
- 与 Django/DRF 相关的特殊 lint 规则

#### 2. `frontend-design` → 合并入 `ui-ux-pro-max`（B 级，6.20 分）

**问题**：仅 43 行，内容单薄，与 `ui-ux-pro-max` 功能重叠（都是前端设计指导）。

**建议**：将 `frontend-design/SKILL.md` 的设计原则（避免 AI 千篇一律、字体配色动效）作为"设计哲学"章节并入 `ui-ux-pro-max`，删除 `frontend-design` 目录。

#### 3. `knowledge-base` Skill → 与 `project-retrospective` 明确边界（B 级，6.75 分）

**问题**：`knowledge-base` Skill（知识读取/沉淀/维护）与 `project-retrospective`（经验沉淀与复用）在"沉淀"功能上有 ~40% 重叠。

**建议**：明确分工——
- `project-retrospective`：**写入**知识库（项目完结时的经验提取与归档）
- `knowledge-base`：**读取+检索**知识库（新项目启动时的经验查询与应用）
- 在两个 Skill 的 description 中互相引用，避免用户触发混淆

#### 4. `django.mdc` → 补充项目特定内容（B 级，6.65 分）

**问题**：115 行偏通用 Django/DRF 规范，缺少 bk-sap-api 项目的特定上下文（如 SAP 集成层约定、OSM 业务模型规范、自定义中间件）。

**建议**：保留通用框架，补充 20-30 行项目特定约定（模型命名约定、SAP 接口封装规范、序列化器继承层级等）。
