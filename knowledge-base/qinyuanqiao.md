## 做得好的地方

### 技术选型

#### 原生开发 + CloudBase

- **选择理由**: MVP 快速验证，免运维，微信能力全兼容
- **实际效果**: 无需关注服务器/域名/SSL，开发效率高
- **推荐指数**: 4/5（限 MVP 阶段）
- **局限**: 云函数冷启动、数据库查询能力有限、不便迁移

### 架构设计

#### 云函数按业务域聚合

- **解决的问题**: 避免每个 API 一个云函数导致管理混乱和冷启动
- **实现方式**: 6 个云函数通过 `action` 参数路由到不同处理函数
- **可复用程度**: 高——微信云开发项目的标准模式
- **复用注意事项**: 单个云函数内的 action 不宜超过 10 个，否则拆分

#### 双模式 UI（CSS 变量主题）

- **解决的问题**: 普通模式和"关照模式"的 UI 差异
- **实现方式**: CSS 变量定义主题色 + 全局状态管理模式标识 + 主题切换函数
- **可复用程度**: 高——CSS 变量 + app.globalData 方式适用于所有小程序主题切换

#### 联系方式安全分级

- **解决的问题**: 用户隐私保护
- **实现方式**: MVP 阶段 Base64 编码，后续计划升级 AES；联系方式仅在审批通过时解密展示
- **可复用程度**: 中——安全策略思路通用，具体加密方案需按场景调整

## 做得不好的地方

### 待改进项

1. **联系方式加密**: Base64 不是加密，MVP 可以但上线前必须换 AES
2. **推荐算法**: 纯规则匹配对用户量大时效果差，需引入协同过滤
3. **内容安全**: 尚未集成微信内容安全 API，涉及合规风险
4. **未上线**: 代码完成但未经过真实用户验证

## 可复用资产

### 代码模式

- **云函数 action 路由模式**: 一个云函数 + switch(action) 处理多个业务操作
- **request.js 统一封装**: 云函数调用统一错误处理 + loading 管理
- **CSS 变量双模式**: 小程序主题切换的轻量方案

### 数据库设计

- **8 集合设计**: users / profiles / contact_requests / members / orders / reports / blacklist / favorites
- **权限策略**: CUSTOM + PRIVATE 组合（用户操作走 CUSTOM 规则校验，敏感操作走 PRIVATE 仅云函数访问）

## 给未来自己的建议

### 如果重新做这个项目

1. 从第一天用 Taro 或 uni-app，保留多端扩展能力
2. 上线前必须集成微信内容安全 API，否则审核不过
3. 支付功能需要企业资质，个人开发者无法使用微信支付

### 延伸到其他项目的通用建议

- **微信云开发适合 MVP**: 快速验证后再考虑是否迁移到自建后端
- **云函数聚合**: 按业务域聚合而非按 API 拆分，减少冷启动
- **小程序审核**: 社交类需要额外资质（如 ICP 备案、增值电信许可），提前规划
