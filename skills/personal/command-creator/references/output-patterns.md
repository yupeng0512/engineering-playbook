## 模板模式

为输出格式提供模板。根据需求选择严格程度。

### 严格模板

适用于 API 响应、数据格式等需要精确一致的场景：

```markdown
## 报告结构

**始终**使用以下精确的模板结构：

# [分析标题]

## 执行摘要
[一段话概述关键发现]

## 关键发现
- 发现 1，附支持数据
- 发现 2，附支持数据
- 发现 3，附支持数据

## 建议
1. 具体可操作的建议
2. 具体可操作的建议
```

### 灵活模板

适用于需要根据情况调整的场景：

```markdown
## 报告结构

以下是合理的默认格式，但请根据情况判断调整：

# [分析标题]

## 执行摘要
[概述]

## 关键发现
[根据发现内容调整章节]

## 建议
[根据具体情况定制]

根据具体分析类型调整章节。
```

## 示例模式

当输出质量依赖于看到示例时，提供输入/输出对：

### 代码提交信息示例

```markdown
## Commit 信息格式

按照以下示例生成 commit 信息：

**示例 1：**
输入：添加了使用 JWT 令牌的用户认证
输出：
```
feat(auth): 实现基于 JWT 的认证

添加登录端点和令牌验证中间件
```

**示例 2：**
输入：修复了报告中日期显示不正确的 bug
输出：
```
fix(reports): 修正时区转换中的日期格式

在报告生成过程中统一使用 UTC 时间戳
```

遵循此风格：类型(范围): 简短描述，然后是详细说明。
```

### 代码审查评论示例

```markdown
## 审查评论格式

按照以下示例撰写审查评论：

**问题类评论：**
```
🔴 **[严重]** 第 42 行
SQL 查询直接拼接用户输入，存在注入风险。
建议使用参数化查询：`cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))`
```

**建议类评论：**
```
🟡 **[建议]** 第 78 行
这个循环可以使用列表推导式简化：
`results = [process(item) for item in items if item.valid]`
```

**赞赏类评论：**
```
🟢 **[赞]** 第 120 行
错误处理写得很好，覆盖了所有边界情况！
```
```

> 💡 **为什么示例有效**：示例帮助 Claude 理解期望的风格和详细程度，比纯文字描述更清晰。

## 结构化输出

### 表格格式

适用于对比、清单、状态报告：

```markdown
## 输出格式

使用表格展示审查结果：

| 文件 | 行号 | 级别 | 问题描述 | 建议 |
|------|------|------|----------|------|
| ... | ... | ... | ... | ... |

级别定义：
- 🔴 严重：必须修复
- 🟠 重要：强烈建议修复
- 🟡 建议：可以改进
- 🟢 信息：仅供参考
```

### JSON 格式

适用于程序化处理的输出：

```markdown
## 输出格式

以 JSON 格式输出分析结果：

```json
{
  "summary": "总体评价",
  "score": 85,
  "issues": [
    {
      "file": "path/to/file.py",
      "line": 42,
      "severity": "high",
      "message": "问题描述",
      "suggestion": "修复建议"
    }
  ],
  "passed_checks": ["check1", "check2"]
}
```
```

### Markdown 报告格式

适用于人类阅读的报告：

```markdown
## 输出格式

生成 Markdown 格式的报告：

# {报告标题}

## 概述
{一段话总结}

## 详细发现

### {类别 1}
{内容}

### {类别 2}
{内容}

## 建议行动
1. {行动 1}
2. {行动 2}

## 附录
{支持数据或参考链接}
```

## 质量标准

### 定义质量标准

在 Skill 中明确定义输出的质量要求：

```markdown
## 质量要求

### 必须满足
- [ ] 所有建议都是可操作的
- [ ] 引用具体的代码行号
- [ ] 提供修复示例（不仅仅是指出问题）

### 应该满足
- [ ] 解释问题的根本原因
- [ ] 考虑边界情况
- [ ] 提供多种解决方案（如适用）

### 避免
- ❌ 模糊的批评（"这段代码不好"）
- ❌ 无依据的建议
- ❌ 过度挑剔微小问题
```

### 输出长度控制

根据场景指定适当的输出长度：

```markdown
## 输出长度

| 场景 | 长度要求 |
|------|----------|
| 快速反馈 | 1-3 句话 |
| 代码审查 | 每个问题 2-5 行 |
| 分析报告 | 500-1000 词 |
| 详细文档 | 根据需要，但有目录 |
```

### 语气和风格

```markdown
## 语气要求

- **专业**：使用技术术语，避免口语化
- **建设性**：指出问题同时提供解决方案
- **简洁**：直接切入重点，避免冗余
- **客观**：基于事实，避免主观评价

示例对比：
❌ "这代码写得很烂，完全看不懂"
✅ "建议重构这个函数：当前 50 行可拆分为 3 个职责单一的小函数，提高可读性"
```

## 组合使用

实际 Skill 通常组合使用多种输出模式：

```markdown
## 输出规范

### 格式
使用 Markdown 格式，包含以下章节：
1. 总体评价（1-2 句话）
2. 问题清单（表格格式）
3. 详细分析（每个重要问题）
4. 建议行动（编号列表）

### 问题清单表格

| 文件 | 行号 | 级别 | 问题 |
|------|------|------|------|

### 详细分析模板

#### {问题标题}
- **位置**：{文件:行号}
- **问题**：{问题描述}
- **影响**：{潜在影响}
- **建议**：{修复方案}
- **示例**：
```{语言}
{修复后的代码示例}
```

### 质量检查
- [ ] 每个问题都有具体行号
- [ ] 每个问题都有修复建议
- [ ] 总体评价与具体发现一致
```
