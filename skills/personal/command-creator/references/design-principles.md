## 简洁优先原则

### 上下文是公共资源

上下文窗口由多个组件共享：
- 系统提示词
- 对话历史
- 其他 Skill 的元数据
- 用户实际请求

**默认假设：Claude 本身已经非常聪明。**

只添加 Claude 尚不具备的上下文。对每条信息提出质疑：
- "Claude 真的需要这个解释吗？"
- "这段内容值得占用这些 token 吗？"

### 简洁的价值

| 冗长方式 | 简洁方式 |
|----------|----------|
| 解释为什么要使用某个库 | 直接给出使用示例 |
| 描述每个步骤的目的 | 提供简明的步骤清单 |
| 列举所有可能的情况 | 给出典型示例 + 边界情况 |

> 💡 **核心准则**：优先使用简洁的示例，而非冗长的解释。

## 自由度匹配原则

将 Claude 想象成在探索一条路径：狭窄的悬崖桥需要具体的护栏（低自由度），而开阔的田野允许多种路线（高自由度）。

### 三级自由度

#### 高自由度（文本指令）

**适用场景**：
- 多种方法都有效
- 决策依赖具体上下文
- 启发式方法引导

**示例**：
```markdown
## 代码审查指南

审查时关注以下方面：
- 代码可读性和清晰度
- 潜在的性能问题
- 安全隐患
- 测试覆盖度

根据具体代码选择重点关注的方面。
```

#### 中等自由度（伪代码/带参数脚本）

**适用场景**：
- 存在推荐模式
- 允许一定变化
- 配置影响行为

**示例**：
```markdown
## 数据处理流程

1. 加载数据：`load_data(source, format=auto)`
2. 验证格式：检查必需字段是否存在
3. 转换处理：
   - 如果是 JSON → 解析为字典
   - 如果是 CSV → 解析为列表
4. 输出结果：`save_data(data, destination)`
```

#### 低自由度（具体脚本/固定步骤）

**适用场景**：
- 操作脆弱易错
- 一致性至关重要
- 必须遵循特定顺序

**示例**：
```markdown
## PDF 表单填写流程

必须严格按以下顺序执行：

1. 运行分析脚本：`python scripts/analyze_form.py input.pdf`
2. 编辑字段映射：修改 `fields.json`
3. 运行验证：`python scripts/validate_fields.py`
4. 填写表单：`python scripts/fill_form.py`
5. 验证输出：`python scripts/verify_output.py`
```

## 渐进式披露原则

### 三级加载机制

Skill 使用三级加载系统高效管理上下文：

| 级别 | 内容 | 加载时机 | 大小建议 |
|------|------|----------|----------|
| **L1** | 元数据（name + description） | 始终在上下文 | ~100 词 |
| **L2** | SKILL.md 正文 | Skill 触发时 | <5k 词（<500 行） |
| **L3** | 捆绑资源 | Claude 按需加载 | 无限制 |

### 渐进式披露模式

#### 模式1：高层指南 + 参考文档

```markdown
# PDF 处理

## 快速开始

使用 pdfplumber 提取文本：
[代码示例]

## 高级功能

- **表单填写**：详见 [references/forms.md](references/forms.md)
- **API 参考**：详见 [references/api.md](references/api.md)
- **示例集合**：详见 [references/examples.md](references/examples.md)
```

Claude 仅在需要时加载 forms.md、api.md 或 examples.md。

#### 模式2：领域特定组织

对于支持多个领域的 Skill，按领域组织内容以避免加载无关上下文：

```
bigquery-skill/
├── SKILL.md（概览和导航）
└── references/
    ├── finance.md（营收、计费指标）
    ├── sales.md（商机、管道）
    ├── product.md（API 使用、功能）
    └── marketing.md（活动、归因）
```

当用户询问销售指标时，Claude 只读取 sales.md。

#### 模式3：框架/变体组织

对于支持多个框架或变体的 Skill：

```
cloud-deploy/
├── SKILL.md（工作流 + 提供商选择）
└── references/
    ├── aws.md（AWS 部署模式）
    ├── gcp.md（GCP 部署模式）
    └── azure.md（Azure 部署模式）
```

当用户选择 AWS 时，Claude 只读取 aws.md。

#### 模式4：条件性详情

显示基本内容，链接到高级内容：

```markdown
# 文档处理

## 创建文档

使用 docx-js 创建新文档。详见 [references/docx-js.md](references/docx-js.md)。

## 编辑文档

简单编辑可直接修改 XML。

**需要跟踪修改？** → 详见 [references/redlining.md](references/redlining.md)
**需要 OOXML 细节？** → 详见 [references/ooxml.md](references/ooxml.md)
```

### 重要指南

1. **避免深度嵌套引用**：所有参考文件应直接从 SKILL.md 链接，保持一层深度
2. **为长文件添加目录**：超过 100 行的参考文件应在顶部包含目录，便于 Claude 预览时了解全貌
3. **避免重复**：信息只存在于 SKILL.md 或参考文件中，不要两者都有

## 工作流模式

### 顺序工作流

对于复杂任务，将操作分解为清晰的顺序步骤。通常在 SKILL.md 开头提供流程概览：

```markdown
填写 PDF 表单涉及以下步骤：

1. 分析表单（运行 analyze_form.py）
2. 创建字段映射（编辑 fields.json）
3. 验证映射（运行 validate_fields.py）
4. 填写表单（运行 fill_form.py）
5. 验证输出（运行 verify_output.py）
```

### 条件工作流

对于有分支逻辑的任务，引导 Claude 通过决策点：

```markdown
1. 确定修改类型：
   **创建新内容？** → 遵循下方"创建工作流"
   **编辑现有内容？** → 遵循下方"编辑工作流"

2. 创建工作流：[步骤]
3. 编辑工作流：[步骤]
```

### 混合工作流

实际 Skill 通常结合多种模式：

```markdown
## 处理流程

1. **输入分析**
   - 确定输入类型
   - 验证格式

2. **处理选择**
   | 输入类型 | 处理方式 |
   |----------|----------|
   | JSON | 解析为字典 |
   | CSV | 解析为列表 |
   | XML | 转换为 JSON |

3. **输出生成**
   根据目标格式选择输出方法
```

## 资源使用最佳实践

### scripts/ 目录

**适合放入**：
- 重复编写的相同代码
- 需要确定性可靠的操作
- 复杂的数据处理逻辑

**注意事项**：
- 脚本可以直接执行而无需加载到上下文
- 但 Claude 仍可能需要读取脚本进行修补或环境调整
- 确保脚本有良好的错误处理和帮助信息

### references/ 目录

**适合放入**：
- 数据库 Schema
- API 文档
- 领域知识
- 公司政策
- 详细工作流指南

**最佳实践**：
- 对于大文件（>10k 词），在 SKILL.md 中包含 grep 搜索模式
- 信息只存在于 SKILL.md 或 references 中，不要重复
- 将详细参考材料、Schema、示例移至 references，保持 SKILL.md 精简

### assets/ 目录

**适合放入**：
- 模板文件（.pptx、.docx）
- 品牌资产（logo、图标）
- 样板代码
- 字体文件
- 示例文档

**特点**：
- 这些文件用于输出，而非加载到上下文
- Claude 可以使用这些文件而无需读取其内容

## 迭代优化

Skill 创建后需要持续迭代优化，以下是经过验证的迭代方法。

### 测试验证

#### 触发测试
- 测试不同表述方式的触发效果
- 验证触发词的覆盖度和精确度
- 检查与其他 Skill 的触发冲突

#### 功能测试
- 验证核心功能是否按预期工作
- 测试边缘情况的处理
- 检查错误处理是否友好

#### 兼容性测试
- 测试不同项目类型的适用性
- 验证跨平台兼容性（如适用）

### 收集反馈

#### 反馈来源
- 自己的实际使用体验
- 团队成员的使用反馈
- 错误日志和失败案例

#### 记录要点
- **问题描述**：具体发生了什么
- **复现步骤**：如何触发问题
- **期望行为**：应该如何表现
- **实际行为**：实际发生了什么

### 持续改进

#### 优化方向

| 方面 | 优化内容 |
|------|----------|
| **触发词** | 扩展覆盖度、减少误触发 |
| **提示词** | 简化表达、提高精确度 |
| **工作流** | 优化步骤、减少摩擦 |
| **示例** | 增加典型案例、补充边界情况 |
| **文档** | 更新过时内容、补充遗漏说明 |

#### 迭代周期建议

```
第一周：频繁使用，收集初始反馈
      ↓
第二周：修复明显问题，调整触发词
      ↓
第一月：根据使用模式优化工作流
      ↓
持续：定期 Review，保持与平台特性同步
```

### 版本管理

#### 变更记录
建议在 Skill 目录下维护 `CHANGELOG.md`（可选）：

```markdown
# 变更记录

## v1.1.0 (2024-01-15)
- 新增：支持 XXX 功能
- 优化：改进触发词覆盖度
- 修复：修复 YYY 场景下的问题

## v1.0.0 (2024-01-01)
- 初始版本发布
```

#### 兼容性考虑
- 避免破坏性变更影响现有用户
- 重大变更需提前通知团队
- 保持向后兼容（如可能）

### 常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| Skill 未触发 | 触发词不匹配 | 在 `description` 中添加更多触发关键词 |
| 触发了错误的 Skill | 触发词冲突 | 调整 `description`，增加区分度 |
| 输出不符合预期 | 提示词不够精确 | 添加更多约束和示例 |
| 执行效率低 | 上下文过大 | 采用渐进式披露 |
| 脚本执行失败 | 环境依赖问题 | 完善错误处理和提示 |
